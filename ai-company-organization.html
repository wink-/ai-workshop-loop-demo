Hermes Multi-Profile Company OS

A proposed operating structure for running a small AI company through Hermes profiles: executive cockpit, senior reasoning, lower-cost implementation lanes, quality gates, release ops, and durable knowledge management.

Routing policy

EscalateUse Senior Engineer when
  • Requirements are ambiguous
  • Architecture/design matters
  • Debugging is tricky
  • A cheaper lane failed
  • Reviewer finds deep design issues
MultimodalUse coder-opencode when
  • Task is well-scoped
  • UI/visual/screenshot context matters
  • Large context helps
  • Cheaper first attempt is desirable
TextUse coder-zai when
  • Task is text/code-only
  • Refactor/docs/repair pass
  • Alternate implementation useful
  • No visual/multimodal understanding needed

Default workflow

CEO/user
  → default cockpit/COO
  → researcher or coder for clarification/spec if needed
  → coder-opencode or coder-zai for scoped implementation
  → reviewer and/or visual-reviewer gates
  → repair pass if needed
  → github-ops release/publish
  → librarian captures durable decisions when useful

Kanban rule: if an implementation card has downstream review children and validation passes, the implementer completes the card. The child review cards are the review gate.
